# Secrets Management at Scale using AWS Secrets Manager

Hardcoding credentials in Kubernetes manifests, passing secrets as plain environment variables, or storing them unencrypted in Git repositories are patterns that lead to breaches. At scale — managing 35+ environments and 100+ microservices — a single leaked credential can cascade into a full production incident.

This guide documents the complete secrets management architecture I built at Mareana: centralized storage in AWS Secrets Manager, automatic rotation, Kubernetes-native injection via External Secrets Operator, and Kyverno policies that prevent any engineer from bypassing the secure path.

---

## Why AWS Secrets Manager Over Kubernetes Secrets?

Kubernetes Secrets are base64-encoded, not encrypted. Anyone with `kubectl get secret` access in a namespace can decode them instantly. They're also stored in etcd — if etcd backups land on S3 without encryption, your secrets are exposed.

AWS Secrets Manager solves this at the storage layer:

| Aspect | Kubernetes Secret | AWS Secrets Manager |
|---|---|---|
| Encryption at rest | ❌ base64 only (unless etcd encryption enabled) | ✅ AES-256 via KMS |
| Access control | K8s RBAC (namespace-scoped) | IAM policies (granular, cross-account) |
| Automatic rotation | ❌ Manual | ✅ Lambda-based, scheduled |
| Audit trail | Limited (K8s audit log) | Full CloudTrail (who accessed, when) |
| Secret versioning | ❌ | ✅ Multiple versions with staging labels |
| Cross-cluster sharing | Manual copy | Single secret, multiple clusters |
| Cost | Free | $0.40/secret/month + $0.05/10K API calls |

For production workloads, the combination of KMS encryption + IAM access control + CloudTrail audit trail makes AWS Secrets Manager the right choice.

## Step 1: Organizing Secrets in AWS Secrets Manager

A consistent naming convention is critical when you have hundreds of secrets:

```
/{environment}/{service}/{secret-type}

prod/payment-service/database
prod/payment-service/stripe-api
prod/payment-service/jwt-signing-key
staging/payment-service/database
dev/payment-service/database
```

Create a secret with structured JSON — one secret can hold multiple related values:

```bash
# Create a database credential secret
aws secretsmanager create-secret \
  --name "prod/payment-service/database" \
  --description "Payment service PostgreSQL credentials" \
  --secret-string '{
    "host":     "payments-db.internal.us-east-1.rds.amazonaws.com",
    "port":     "5432",
    "username": "payments_app",
    "password": "$(openssl rand -base64 32)",
    "dbname":   "payments_prod"
  }' \
  --tags '[
    {"Key": "Environment", "Value": "prod"},
    {"Key": "Service",     "Value": "payment-service"},
    {"Key": "Team",        "Value": "billing-team"},
    {"Key": "ManagedBy",   "Value": "platform-team"}
  ]'
```

Use resource-based policies to restrict access by service:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789:role/payment-service-pod-role"
      },
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:123456789:secret:prod/payment-service/*"
    }
  ]
}
```

This enforces least privilege at the secret level — the payment service role can only access payment service secrets.

## Step 2: IAM Roles for Service Accounts (IRSA)

IRSA links a Kubernetes ServiceAccount to an IAM role, allowing pods to authenticate to AWS without any static credentials:

```bash
# Create the IAM role for the payment service
aws iam create-role \
  --role-name payment-service-pod-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B716D3041E"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B716D3041E:sub":
            "system:serviceaccount:prod-payments:payment-service-sa"
        }
      }
    }]
  }'

# Attach the secrets access policy
aws iam attach-role-policy \
  --role-name payment-service-pod-role \
  --policy-arn arn:aws:iam::123456789:policy/payment-service-secrets-policy
```

```yaml
# Kubernetes ServiceAccount annotated with the IAM role
apiVersion: v1
kind: ServiceAccount
metadata:
  name: payment-service-sa
  namespace: prod-payments
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/payment-service-pod-role
```

## Step 4: ClusterSecretStore — AWS Provider

```yaml
# cluster-secret-store-aws.yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secretsmanager
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets
```

The `ClusterSecretStore` is cluster-wide — all namespaces can reference it. For stricter multi-tenancy, create namespace-scoped `SecretStore` resources per team.

## Step 5: ExternalSecret — Syncing Secrets

```yaml
# payment-service-external-secrets.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-db-secret
  namespace: prod-payments
spec:
  # Refresh every hour — picks up rotated passwords automatically
  refreshInterval: 1h

  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore

  target:
    name: payment-db-credentials   # Name of the K8s Secret to create
    creationPolicy: Owner           # ESO owns the secret; deleting ExternalSecret deletes the K8s Secret
    template:
      # Format the secret exactly how the app expects it
      data:
        DATABASE_URL: "postgresql://{{ .username }}:{{ .password }}@{{ .host }}:{{ .port }}/{{ .dbname }}"
        DB_HOST:     "{{ .host }}"
        DB_PORT:     "{{ .port }}"
        DB_USER:     "{{ .username }}"
        DB_PASSWORD: "{{ .password }}"
        DB_NAME:     "{{ .dbname }}"

  data:
    # Pull all fields from a single JSON secret
    - secretKey: host
      remoteRef:
        key: prod/payment-service/database
        property: host
    - secretKey: port
      remoteRef:
        key: prod/payment-service/database
        property: port
    - secretKey: username
      remoteRef:
        key: prod/payment-service/database
        property: username
    - secretKey: password
      remoteRef:
        key: prod/payment-service/database
        property: password
    - secretKey: dbname
      remoteRef:
        key: prod/payment-service/database
        property: dbname
```

Or use `dataFrom` to pull the entire JSON object at once:

```yaml
  dataFrom:
    - extract:
        key: prod/payment-service/database
      # Pulls all JSON fields as separate env var keys
```

## Step 6: Using the Secret in a Pod

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
  namespace: prod-payments
spec:
  template:
    spec:
      serviceAccountName: payment-service-sa
      containers:
        - name: payment-service
          image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/naik-prod/payment-service:v5.1.0
          envFrom:
            # Inject ALL keys from the synced secret as env vars
            - secretRef:
                name: payment-db-credentials
          # Or inject specific keys
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: payment-db-credentials
                  key: DATABASE_URL
```

The application receives the secret value as a standard environment variable — it has no awareness of AWS Secrets Manager at all.

## Step 7: Automatic Secret Rotation

Configure AWS Secrets Manager to rotate the PostgreSQL password automatically every 30 days using the managed Lambda rotator:

```bash
# Enable automatic rotation with AWS-managed Lambda
aws secretsmanager rotate-secret \
  --secret-id "prod/payment-service/database" \
  --rotation-lambda-arn "arn:aws:lambda:us-east-1:123456789:function:SecretsManagerRDSPostgreSQLRotationSingleUser" \
  --rotation-rules AutomaticallyAfterDays=30

# Trigger an immediate rotation (useful for testing)
aws secretsmanager rotate-secret \
  --secret-id "prod/payment-service/database" \
  --rotate-immediately
```

**How rotation + ESO work together:**
1. AWS rotates the password (updates secret version in Secrets Manager)
2. ESO's `refreshInterval: 1h` polls the secret hourly
3. ESO detects the version change and updates the Kubernetes Secret
4. Pods using `envFrom` pick up the new value on next restart — or immediately if using CSI driver mounts

For zero-downtime rotation, use the **AWS Secrets Manager + Kubernetes CSI driver** approach where secrets are mounted as files that update in-place:

```yaml
# Using CSI driver for live-updating secret mounts
spec:
  volumes:
    - name: db-creds
      csi:
        driver: secrets-store.csi.k8s.io
        readOnly: true
        volumeAttributes:
          secretProviderClass: payment-db-provider
  containers:
    - name: payment-service
      volumeMounts:
        - name: db-creds
          mountPath: /mnt/secrets
          readOnly: true
```

## Step 9: Kyverno Policy — Enforce ESO-Only Secrets

This Kyverno policy prevents anyone from creating raw `kubectl create secret` objects in production namespaces — all secrets must come through ESO:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-eso-managed-secrets
spec:
  validationFailureAction: Enforce
  background: false
  rules:
    - name: check-secret-owner
      match:
        any:
          - resources:
              kinds: [Secret]
              namespaces: ["prod-*", "staging-*"]
      exclude:
        any:
          # Allow system-managed secrets
          - resources:
              kinds: [Secret]
              names: ["default-token-*", "sh.helm.release.*"]
          # Allow secrets in system namespaces
          - resources:
              namespaces: [kube-system, kyverno, external-secrets, flux-system]
      validate:
        message: >
          Production secrets must be managed by External Secrets Operator.
          Use ExternalSecret CRD to sync from AWS Secrets Manager.
          Contact platform-team if you need an exception.
        deny:
          conditions:
            all:
              - key: "{{ request.object.metadata.ownerReferences | length(@) }}"
                operator: Equals
                value: "0"
```

Testing the policy works:

```bash
# This is BLOCKED
kubectl create secret generic test-secret \
  --from-literal=password=abc123 \
  -n prod-payments

# Error: admission webhook "validate.kyverno.svc" denied the request:
# Production secrets must be managed by External Secrets Operator.
```

## Key Takeaways

1. **Treat secrets like code** — centralized, versioned, audited, and access-controlled
2. **IRSA is the right auth pattern for EKS** — no static credentials ever leave AWS
3. **`refreshInterval` must be shorter than your rotation window** — if you rotate every 30 days, refresh every hour
4. **Use `dataFrom` for simple cases, `data` for formatting** — the `template` block lets you reshape secrets exactly how apps expect them
5. **Kyverno enforcement is the last line of defense** — without it, one `kubectl create secret` bypasses the entire architecture
