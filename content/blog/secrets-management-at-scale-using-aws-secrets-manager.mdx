# Secrets Management at Scale using AWS Secrets Manager

Hardcoding database passwords, API tokens, and access keys in deployment files is a massive security risk. 

For Kubernetes architectures, we need a secure, auditable, and automated secrets lifecycle management system. In this guide, we'll demonstrate how to sync and inject secrets from **AWS Secrets Manager** using the External Secrets Operator.

---

## 1. Secrets Management Lifecycle

An enterprise secrets architecture requires:
- **Centralized Storage**: Enforcing IAM permissions, access audits, and KMS decryption keys.
- **Dynamic Rotation**: Periodically rotating passwords without causing application downtime.
- **Secure Injection**: Exposing secrets only in memory (as environment variables or tmpfs files) inside containers.

---

## 2. Syncing Secrets with External Secrets Operator

The **External Secrets Operator (ESO)** is a Kubernetes operator that fetches secrets from external managers (like AWS Secrets Manager, Azure Key Vault, or HashiCorp Vault) and injects them as native K8s `Secret` resources.

### Step A: Define the SecretStore provider

This tells Kubernetes how to authenticate with the AWS Secrets Manager service:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-secrets-store
  namespace: prod-ems
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: eso-irsa-service-account # Authenticates via AWS IAM Roles for Service Accounts (IRSA)
```

---

## 3. Creating the ExternalSecret Mapping

Now we declare a mapping which retrieves a specific secret key from AWS and writes it into a Kubernetes secret:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: ems-database-secret
  namespace: prod-ems
spec:
  refreshInterval: "1h" # Automatically syncs rotated passwords from AWS every hour
  secretStoreRef:
    name: aws-secrets-store
    kind: SecretStore
  target:
    name: ems-db-secret # Target Kubernetes Secret Name
    creationPolicy: Owner
  data:
    - secretKey: db-password # Secret Key inside Kubernetes Secret
      remoteRef:
        key: prod/ems/database # Remote AWS Secrets Manager secret name
        property: password    # Key inside JSON secret block
```

---

## 4. Querying Secret Sync Status

To verify that the External Secrets Operator has successfully synced the passwords, run:

```bash
kubectl describe externalsecret ems-database-secret -n prod-ems
```

This will report the sync state and indicate if the secret was successfully created and decrypted inside the cluster namespace.
