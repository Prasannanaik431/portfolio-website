# Building Enterprise GitOps with Flux2

GitOps has become the industry standard for declaring and reconciling Kubernetes cluster states. By treating Git as the single source of truth, teams can maintain auditable, declarative, and automated rollouts. 

In this article, we'll walk through designing a robust, multi-tenant GitOps model using **Flux2**.

---

## 1. Directory Structure

A clean repository structure separates cluster configurations, infrastructure operators, and application workloads:

```bash
├── clusters
│   ├── production
│   │   ├── flux-system/         # Autogenerated Flux system components
│   │   ├── infrastructure.yaml  # Points to infra/ Kustomization
│   │   └── apps.yaml            # Points to apps/ Kustomization
├── infra
│   ├── base
│   │   ├── prometheus/          # Prometheus operator helm manifests
│   │   └── ingress-nginx/       # Nginx Ingress helm manifests
│   └── production
│       └── patches/             # Environment overrides
└── apps
    ├── staging
    └── production
        ├── ems-service.yaml     # Application HelmRelease manifest
        └── idp-service.yaml
```

---

## 2. Defining the GitRepository Source

The core Flux resource is the `GitRepository`, which tells the Flux Source Controller how to fetch cluster specifications:

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 1m0s
  ref:
    branch: main
  secretRef:
    name: flux-system-auth
  url: ssh://git@github.com/naik-org/ems-infra.git
```

---

## 3. Implementing Kustomization Reconcilers

To apply resources from Git to the cluster, we use `Kustomization` reconcilers. Here's how we map the infrastructure tier:

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cluster-infra
  namespace: flux-system
spec:
  interval: 5m0s
  path: ./infra/production
  prune: true # Delete deleted Git resources from cluster automatically
  sourceRef:
    kind: GitRepository
    name: flux-system
  timeout: 2m0s
  wait: true # Wait for resources to be healthy before completing sync
```

## 4. Deploying Applications via HelmRelease

Flux integrates natively with Helm via the `HelmRelease` resource. This enables declarative application rollouts using Helm charts directly from registries or git:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2beta1
kind: HelmRelease
metadata:
  name: ems-service
  namespace: prod-ems
spec:
  interval: 5m
  chart:
    spec:
      chart: ems-service
      version: "1.2.0"
      sourceRef:
        kind: HelmRepository
        name: naik-helm-charts
        namespace: flux-system
  install:
    remediation:
      retries: 3
  upgrade:
    remediation:
      retries: 3
  values:
    replicaCount: 3
    resources:
      limits:
        cpu: 1000m
        memory: 1024Mi
      requests:
        cpu: 200m
        memory: 256Mi
    ingress:
      enabled: true
      hosts:
        - host: ems.naik.internal
          paths:
            - path: /
```

---

## 5. Reconciling Git changes manually

To verify or force Flux synchronization immediately without waiting for the interval, run:

```bash
flux reconcile kustomization ems-app --with-source
```

This ensures instant rollouts during development or hotfix actions. In the next guide, we will look at automating secrets insertion into these templates.
