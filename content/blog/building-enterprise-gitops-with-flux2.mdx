# Building Enterprise GitOps with Flux2

GitOps has become the industry standard for declaring and reconciling Kubernetes cluster states. By treating Git as the single source of truth, teams can maintain auditable, declarative, and automated rollouts — with every change traceable to a commit.

At Mareana, we run 35+ environments across AWS EKS and Azure AKS. Before GitOps, each environment was a snowflake — manually applied YAML, no audit trail, no rollback, and config drift within days of provisioning. After migrating to Flux2, every environment is a faithful reflection of a Git branch. Deployments are automated, drift is detected and corrected within minutes, and rollbacks are a one-line git revert.

---

## What is GitOps — and Why Does It Matter?

GitOps is an operational model where:
1. **Desired state is declared in Git** — all Kubernetes manifests, Helm values, and Kustomize patches live in a repository
2. **An agent reconciles continuously** — Flux2 constantly compares the live cluster state against Git and converges any differences
3. **Changes happen via pull requests** — no `kubectl apply` in production; all changes go through code review

The practical benefits at scale:
- **Auditability**: `git log` is your deployment log. Every change has an author, timestamp, and PR context.
- **Disaster recovery**: Re-provision a cluster from scratch by pointing Flux at the same repository.
- **Drift prevention**: Config drift is automatically corrected — someone manually patching a pod's CPU limits? Flux reverts it within minutes.
- **Multi-environment parity**: Staging and production diverge only where you explicitly configure them to.

The key insight: **Flux never pushes from outside the cluster**. It runs inside the cluster and pulls from Git. This means no inbound firewall rules needed and no external CI system needs cluster credentials.

## How Flux2 Differs from ArgoCD

| Feature | Flux2 | ArgoCD |
|---|---|---|
| UI | No built-in UI | Rich web UI |
| Multi-tenancy | Native (per-namespace controllers) | Requires careful RBAC setup |
| Kustomize support | Native, first-class | Supported |
| Helm support | Native HelmRelease CRD | Native Application CRD |
| Image automation | Built-in controller | Requires Argo Image Updater plugin |
| Architecture | Decentralised (per-cluster controllers) | Central server + agents |
| Learning curve | Higher (YAML-heavy) | Lower (UI-guided) |

For large multi-cluster setups where teams want full GitOps purity and no UI dependency, Flux2 is the better choice. ArgoCD is better when you need a visual dashboard for a single-cluster setup.

## 1. Repository Structure

A clean repository structure is the foundation of a scalable GitOps model. The key principle: **separate cluster bootstrap config from application config from infrastructure config**.

```bash
├── clusters/
│   ├── production/
│   │   ├── flux-system/         # Auto-generated by flux bootstrap
│   │   ├── infrastructure.yaml  # Kustomization → infra/production
│   │   └── apps.yaml            # Kustomization → apps/production
│   └── staging/
│       ├── flux-system/
│       ├── infrastructure.yaml
│       └── apps.yaml
├── infra/
│   ├── base/
│   │   ├── prometheus/          # kube-prometheus-stack HelmRelease
│   │   ├── ingress-nginx/       # nginx HelmRelease
│   │   └── external-secrets/   # ESO operator HelmRelease
│   ├── production/
│   │   └── patches/             # Production-specific overrides
│   └── staging/
│       └── patches/             # Staging overrides (smaller resources)
└── apps/
    ├── base/
    │   └── ems-service/         # Base HelmRelease template
    ├── production/
    │   └── ems-service.yaml     # Prod values override
    └── staging/
        └── ems-service.yaml     # Staging values override
```

**Why this structure?**

- `clusters/` is the entry point — Flux bootstraps here and discovers everything else
- `infra/` contains cluster-level operators (Prometheus, Nginx, ESO) — deployed before apps
- `apps/` contains application workloads — deployed after infra is healthy
- `base/` holds environment-agnostic defaults; environment folders hold overrides

This separation means you can onboard a new cluster by creating a new folder in `clusters/` and pointing Flux at it — everything else deploys automatically.

## 2. Bootstrapping Flux

Install Flux CLI and bootstrap it against your cluster. This creates the `flux-system` namespace and commits the controllers to your Git repository:

```bash
# Install Flux CLI
curl -s https://fluxcd.io/install.sh | sudo bash

# Verify cluster prerequisites
flux check --pre

# Bootstrap Flux — this commits controllers to Git and installs them
flux bootstrap github \
  --owner=naik-org \
  --repository=ems-infra \
  --branch=main \
  --path=clusters/production \
  --personal=false \
  --token-auth

# Verify all controllers are running
flux check

# NAME                                       READY  MESSAGE
# source-controller                          True   Deployment is ready
# kustomize-controller                       True   Deployment is ready
# helm-controller                            True   Deployment is ready
# notification-controller                    True   Deployment is ready
# image-automation-controller                True   Deployment is ready
# image-reflector-controller                 True   Deployment is ready
```

After bootstrap, Flux is watching `clusters/production/` in Git. Any YAML you push there is automatically applied.

## 3. Defining the GitRepository Source

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 1m0s        # Poll Git every minute
  ref:
    branch: main
  secretRef:
    name: flux-system-auth   # SSH deploy key stored as K8s Secret
  url: ssh://git@github.com/naik-org/ems-infra.git
  # Ignore non-relevant file changes to avoid unnecessary reconciliations
  ignore: |
    # Ignore docs, CI configs, and test files
    /docs/
    /.github/
    /**/*_test.yaml
```

The `ignore` field is important at scale — without it, updating a README triggers a full cluster reconciliation.

## 4. Kustomization Reconcilers — Ordering Matters

Flux applies resources in dependency order. Infrastructure (Prometheus, Nginx) must be healthy before apps deploy:

```yaml
# clusters/production/infrastructure.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cluster-infra
  namespace: flux-system
spec:
  interval: 10m
  path: ./infra/production
  prune: true           # Auto-delete resources removed from Git
  sourceRef:
    kind: GitRepository
    name: flux-system
  timeout: 5m
  wait: true            # Block until all resources are healthy
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: ingress-nginx-controller
      namespace: ingress-nginx
---
# clusters/production/apps.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cluster-apps
  namespace: flux-system
spec:
  interval: 5m
  path: ./apps/production
  prune: true
  sourceRef:
    kind: GitRepository
    name: flux-system
  dependsOn:
    - name: cluster-infra  # Apps only deploy AFTER infra is healthy
  timeout: 3m
```

The `dependsOn` field is critical — it ensures Nginx and Prometheus are running before services try to register their `Ingress` resources.

## 5. Deploying Applications via HelmRelease

```yaml
# apps/production/ems-service.yaml
apiVersion: helm.toolkit.fluxcd.io/v2beta1
kind: HelmRelease
metadata:
  name: ems-service
  namespace: prod-ems
spec:
  interval: 5m
  chart:
    spec:
      chart: app-template
      version: "2.1.0"
      sourceRef:
        kind: HelmRepository
        name: naik-helm-charts
        namespace: flux-system
  valuesFrom:
    # Base values common to all environments
    - kind: ConfigMap
      name: ems-service-base-values
    # Production-specific overrides
    - kind: ConfigMap
      name: ems-service-prod-values
      optional: false
  install:
    remediation:
      retries: 3
  upgrade:
    remediation:
      retries: 3
      remediateLastFailure: true
    cleanupOnFail: true    # Delete failed upgrade resources
  rollback:
    timeout: 5m
    cleanupOnFail: true
  values:
    replicaCount: 3
    resources:
      limits:   { cpu: 1000m, memory: 1Gi }
      requests: { cpu: 200m,  memory: 256Mi }
```

## 6. Image Automation — Zero-Touch Deployments

With image automation, Flux detects new container image tags and automatically opens a commit to update the deployment — no CI system needs cluster access:

```yaml
# flux-system/image-repository-ems.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: ems-service
  namespace: flux-system
spec:
  image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/naik-prod/ems-service
  interval: 5m
  secretRef:
    name: gcr-pull-secret
---
# Only auto-deploy patch versions (e.g. v2.1.x) — never major bumps
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: ems-service
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: ems-service
  policy:
    semver:
      range: ">=2.1.0 <2.2.0"
---
# Write the new tag back to the values file in Git
apiVersion: image.toolkit.fluxcd.io/v1beta1
kind: ImageUpdateAutomation
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: flux-system
  git:
    checkout:
      ref:
        branch: main
    commit:
      author:
        name: flux-bot
        email: flux@naik.internal
      messageTemplate: |
        chore(deploy): update {{range .Updated.Images}}{{.}}{{end}} [skip ci]
    push:
      branch: main
  update:
    strategy: Setters   # Updates YAML files with # {"$imagepolicy": "flux-system:ems-service"} markers
```

In the `values.yaml`, add the marker so Flux knows which field to update:

```yaml
image:
  repository: 123456789012.dkr.ecr.us-east-1.amazonaws.com/naik-prod/ems-service
  tag: v2.1.4 # {"$imagepolicy": "flux-system:ems-service:tag"}
```

## 7. Multi-Tenancy: Per-Team Namespaces

In a multi-tenant setup, each team gets their own Flux Kustomization scoped to their namespace — they can't accidentally deploy to another team's namespace:

```yaml
# Team billing gets their own Kustomization
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: team-billing-apps
  namespace: flux-system
spec:
  serviceAccountName: flux-billing-reconciler  # Scoped service account
  interval: 5m
  path: ./teams/billing/apps
  prune: true
  sourceRef:
    kind: GitRepository
    name: flux-system
  targetNamespace: prod-billing   # Force all resources into this namespace
```

The `targetNamespace` field prevents teams from deploying to arbitrary namespaces. Combined with Kyverno policies, this enforces strong multi-tenancy.

## 8. Monitoring Flux Health

Flux exposes Prometheus metrics. Key alerts to configure:

```yaml
# Alert when Flux can't reconcile for > 10 minutes
- alert: FluxReconciliationFailed
  expr: |
    gotk_reconcile_condition{type="Ready", status="False"} == 1
  for: 10m
  labels:
    severity: warning
  annotations:
    summary: "Flux reconciliation failing: {{ $labels.name }}"
    description: "{{ $labels.kind }} {{ $labels.name }} in {{ $labels.namespace }} has been failing to reconcile."

# Alert when a HelmRelease is degraded
- alert: HelmReleaseDegraded
  expr: |
    gotk_reconcile_condition{kind="HelmRelease", type="Released", status="False"} == 1
  for: 5m
  labels:
    severity: critical
```

## 9. Forcing Immediate Reconciliation

```bash
# Force sync a specific Kustomization
flux reconcile kustomization cluster-apps --with-source

# Force sync a HelmRelease
flux reconcile helmrelease ems-service -n prod-ems

# Suspend reconciliation (for maintenance)
flux suspend kustomization cluster-apps

# Resume
flux resume kustomization cluster-apps

# View reconciliation history
flux get all -A

# Debug a failing HelmRelease
flux logs --kind=HelmRelease --name=ems-service -n prod-ems --follow
```

## Key Takeaways

1. **Start with the directory structure** — getting the `clusters/` + `infra/` + `apps/` separation right prevents painful refactoring later
2. **Use `dependsOn` religiously** — apps should never deploy before their dependencies are healthy
3. **`prune: true` is powerful but dangerous** — always test `prune` in staging first; a bad commit can delete production resources
4. **Image automation is the real productivity win** — eliminating manual image tag bumps saves hours per week
5. **Monitor Flux itself** — add Prometheus alerts for reconciliation failures; silent drift is worse than loud failures
