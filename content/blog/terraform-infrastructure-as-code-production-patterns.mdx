# Terraform IaC Production Patterns: Modules, Remote State & Multi-Cloud

After managing 35+ production environments across AWS and Azure, I've learned that raw Terraform is only half the story. The difference between Terraform scripts that work once and Terraform that runs reliably at enterprise scale comes down to three things: **module architecture**, **state management**, and **pipeline integration**.

This is an opinionated guide to what actually works in production.

---

## Why Terraform at Scale Gets Painful

Most teams start with a flat `main.tf` that works fine for one environment. The problems start when you have 10+ environments, multiple teams, and cloud-spanning infrastructure:

- State file contention when two engineers run `apply` simultaneously
- Copy-pasted resource blocks with subtle drift between environments
- No clear boundary between who owns which resources
- Unreviewed `plan` outputs that silently destroy production load balancers

The patterns below solve each of these.

## 1. Module Hierarchy: Three-Layer Pattern

<ArchitectureDiagram />

> **Architecture at a glance:** Primitive modules know nothing about environments. Compositions wire primitives; environment folders only pass `tfvars`. Remote state in S3 with DynamoDB locking prevents concurrent apply races.

Structure your Terraform code into three layers. This is the single most impactful architectural decision.

```
infra/
├── modules/               # Layer 1: Reusable primitives
│   ├── eks-cluster/
│   ├── rds-postgres/
│   ├── vpc-network/
│   └── aks-nodepool/
├── compositions/          # Layer 2: Environment blueprints
│   ├── aws-saas-stack/
│   └── azure-saas-stack/
└── environments/          # Layer 3: Live environment configs
    ├── prod-us-east/
    ├── staging-eu-west/
    └── dev-sandbox/
```

**Layer 1 — Primitive Modules**: Single-responsibility, no environment awareness. An EKS module just creates an EKS cluster and outputs the endpoint. Nothing else.

**Layer 2 — Compositions**: Wire together multiple primitives for a full stack. The `aws-saas-stack` composition calls `vpc-network`, `eks-cluster`, and `rds-postgres` modules.

**Layer 3 — Environments**: Thin wrappers that pass environment-specific `tfvars` into compositions. This is where you define cluster sizes, region, tags, and cost allocations.

### Example: EKS Cluster Primitive Module

```hcl
# modules/eks-cluster/main.tf

variable "cluster_name"    { type = string }
variable "k8s_version"     { type = string  default = "1.35" }
variable "node_instance_type" { type = string  default = "t3.medium" }
variable "desired_nodes"   { type = number  default = 3 }
variable "min_nodes"       { type = number  default = 1 }
variable "max_nodes"       { type = number  default = 10 }
variable "subnet_ids"      { type = list(string) }
variable "tags"            { type = map(string)  default = {} }

resource "aws_eks_cluster" "this" {
  name     = var.cluster_name
  version  = var.k8s_version
  role_arn = aws_iam_role.cluster.arn

  vpc_config {
    subnet_ids              = var.subnet_ids
    endpoint_public_access  = false
    endpoint_private_access = true
  }

  tags = merge(var.tags, { ManagedBy = "terraform" })

  depends_on = [aws_iam_role_policy_attachment.cluster_policy]
}

resource "aws_eks_node_group" "default" {
  cluster_name    = aws_eks_cluster.this.name
  node_group_name = "${var.cluster_name}-default"
  node_role_arn   = aws_iam_role.node.arn
  subnet_ids      = var.subnet_ids
  instance_types  = [var.node_instance_type]

  scaling_config {
    desired_size = var.desired_nodes
    min_size     = var.min_nodes
    max_size     = var.max_nodes
  }

  update_config {
    max_unavailable = 1
  }

  tags = var.tags
}

output "cluster_endpoint"        { value = aws_eks_cluster.this.endpoint }
output "cluster_name"            { value = aws_eks_cluster.this.name }
output "cluster_certificate_authority" { value = aws_eks_cluster.this.certificate_authority[0].data }
```

### Example: Environment Layer (Thin Wrapper)

```hcl
# environments/prod-us-east/main.tf

terraform {
  required_version = ">= 1.8.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "= 5.55.0"   # Pin exact version in production
    }
  }

  backend "s3" {
    bucket         = "naik-terraform-state-prod"
    key            = "prod-us-east/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
  }
}

module "saas_stack" {
  source = "../../compositions/aws-saas-stack"

  environment        = "prod"
  region             = "us-east-1"
  cluster_node_type  = "m5.xlarge"
  desired_nodes      = 6
  min_nodes          = 3
  max_nodes          = 15
  db_instance_class  = "db.r6g.xlarge"
  db_storage_gb      = 200
  
  tags = {
    Environment = "prod"
    Owner       = "platform-team"
    CostCenter  = "engineering"
    Project     = "ems-saas"
  }
}
```

## 2. Remote State: S3 + DynamoDB Locking

Never use local state in a team environment. S3 backend with DynamoDB locking prevents simultaneous apply conflicts.

### Bootstrap the State Bucket (one-time)

```hcl
# bootstrap/state-backend.tf
# Run this ONCE manually with local state, then migrate

resource "aws_s3_bucket" "terraform_state" {
  bucket = "naik-terraform-state-prod"

  lifecycle {
    prevent_destroy = true  # Never auto-delete the state bucket
  }
}

resource "aws_s3_bucket_versioning" "state_versioning" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration {
    status = "Enabled"  # Allows rollback to previous state versions
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "state_encryption" {
  bucket = aws_s3_bucket.terraform_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_dynamodb_table" "state_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}
```

### Reading Shared State with `data.terraform_remote_state`

When one stack needs outputs from another (e.g., the app stack needs the VPC ID from the network stack), use remote state data sources:

```hcl
# In app stack: read the network stack's outputs
data "terraform_remote_state" "network" {
  backend = "s3"
  config = {
    bucket = "naik-terraform-state-prod"
    key    = "prod-us-east/network/terraform.tfstate"
    region = "us-east-1"
  }
}

module "eks" {
  source     = "../../modules/eks-cluster"
  subnet_ids = data.terraform_remote_state.network.outputs.private_subnet_ids
}
```

## 3. Multi-Cloud: AWS + Azure in One Pipeline

At Mareana we run workloads on both AWS EKS and Azure (AKS). Using provider aliasing, a single Terraform workspace can manage resources across both clouds.

```hcl
# Provider configuration with aliases
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "= 5.55.0"
    }
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "= 3.110.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
  alias  = "primary"
  assume_role {
    role_arn = "arn:aws:iam::123456789:role/TerraformRole"
  }
}

provider "azurerm" {
  alias           = "azure_eu"
  subscription_id = var.azure_subscription_id
  features {}
}
```

### Sharing Secrets Across Clouds

A common pattern: store an Azure SPN secret in AWS Secrets Manager so Terraform workflows pulling Azure credentials don't need static environment variables.

```hcl
data "aws_secretsmanager_secret_version" "azure_spn" {
  provider  = aws.primary
  secret_id = "platform/azure/service-principal"
}

locals {
  azure_creds = jsondecode(data.aws_secretsmanager_secret_version.azure_spn.secret_string)
}

provider "azurerm" {
  alias           = "azure_prod"
  subscription_id = local.azure_creds.subscription_id
  client_id       = local.azure_creds.client_id
  client_secret   = local.azure_creds.client_secret
  tenant_id       = local.azure_creds.tenant_id
  features {}
}
```

## 4. Workspace Strategy for Environment Isolation

Terraform workspaces let you share the same configuration across environments. We use a workspace-per-environment model.

```bash
# Create and switch to staging workspace
terraform workspace new staging
terraform workspace select staging

# Apply using workspace-specific tfvars
terraform apply -var-file="environments/staging.tfvars"
```

```hcl
# Use workspace name to drive dynamic configuration
locals {
  workspace_config = {
    dev     = { node_count = 1, instance_type = "t3.medium",  db_class = "db.t3.micro"   }
    staging = { node_count = 2, instance_type = "t3.large",   db_class = "db.t3.medium"  }
    prod    = { node_count = 6, instance_type = "m5.xlarge",  db_class = "db.r6g.xlarge" }
  }

  config = local.workspace_config[terraform.workspace]
}

module "eks" {
  source           = "../../modules/eks-cluster"
  desired_nodes    = local.config.node_count
  node_instance_type = local.config.instance_type
}
```

## 5. Pipeline Integration: Plan + Apply with PR Review

Never run `terraform apply` manually in production. Every change flows through a CI pipeline with a mandatory plan review step.

```groovy
// Jenkins pipeline — Terraform CD
pipeline {
  agent { label 'terraform-agent' }

  environment {
    TF_VAR_environment   = "${params.ENVIRONMENT}"
    AWS_ROLE_ARN         = credentials('terraform-role-arn')
  }

  stages {
    stage('Init') {
      steps {
        sh '''
          cd environments/${TF_VAR_environment}
          terraform init -reconfigure -backend-config="key=${TF_VAR_environment}/terraform.tfstate"
        '''
      }
    }

    stage('Validate') {
      steps {
        sh '''
          terraform validate
          terraform fmt -check -recursive
        '''
      }
    }

    stage('Plan') {
      steps {
        sh '''
          cd environments/${TF_VAR_environment}
          terraform plan -out=tfplan -detailed-exitcode
        '''
        // Save plan artifact for apply stage
        archiveArtifacts artifacts: 'environments/**/tfplan'
      }
    }

    stage('Approval') {
      when { branch 'main' }
      steps {
        input message: "Review Terraform plan for ${TF_VAR_environment}. Proceed with apply?",
              ok: "Apply Changes"
      }
    }

    stage('Apply') {
      when { branch 'main' }
      steps {
        sh '''
          cd environments/${TF_VAR_environment}
          terraform apply -auto-approve tfplan
        '''
      }
    }
  }

  post {
    always {
      cleanWs()
    }
    failure {
      slackSend channel: '#platform-alerts', color: 'danger',
                message: "Terraform FAILED for ${TF_VAR_environment}: ${env.BUILD_URL}"
    }
  }
}
```

## 6. Drift Detection with Scheduled Plan Runs

Infrastructure drift happens silently. A scheduled Jenkins job runs `terraform plan` daily on all environments and alerts if the plan is non-empty.

```bash
#!/bin/bash
# drift-check.sh — Run in a daily scheduled pipeline

ENVIRONMENTS=("prod-us-east" "staging-eu-west" "dev-sandbox")
DRIFT_FOUND=0

for env in "${ENVIRONMENTS[@]}"; do
  echo "Checking drift in: $env"
  cd "environments/$env"
  terraform init -reconfigure > /dev/null 2>&1

  # -detailed-exitcode: 0=no changes, 1=error, 2=changes present
  terraform plan -detailed-exitcode -out=/dev/null
  EXIT_CODE=$?

  if [ $EXIT_CODE -eq 2 ]; then
    echo "⚠️  DRIFT DETECTED in $env"
    DRIFT_FOUND=1
    # Send Slack notification
    curl -s -X POST "$SLACK_WEBHOOK" \
      -H 'Content-type: application/json' \
      -d "{\"text\": \"⚠️ Terraform drift detected in \`$env\`. Manual review required.\"}"
  fi

  cd - > /dev/null
done

exit $DRIFT_FOUND
```

## 7. Secrets Injection: Never Store Secrets in State

Terraform state files are stored in S3 — but state contains every resource attribute including sensitive values. Use AWS Secrets Manager + `sensitive = true` to limit exposure.

```hcl
# Fetch DB password from Secrets Manager at plan time
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "platform/${var.environment}/db-password"
}

resource "aws_db_instance" "postgres" {
  identifier     = "ems-${var.environment}-db"
  engine         = "postgres"
  engine_version = "15.4"
  instance_class = var.db_instance_class
  
  # Password fetched from Secrets Manager — never in source code
  password       = data.aws_secretsmanager_secret_version.db_password.secret_string
  username       = "ems_admin"

  # Ensure password doesn't appear in plan output
  lifecycle {
    ignore_changes = [password]
  }
}

output "db_endpoint" {
  value     = aws_db_instance.postgres.endpoint
  sensitive = false  # Endpoint is safe to expose
}

output "db_password_arn" {
  value     = data.aws_secretsmanager_secret_version.db_password.arn
  sensitive = true  # Mark as sensitive — won't show in plan output
}
```

## Which Modular Approach Should You Choose?

There are several module architecture patterns in the Terraform ecosystem. Here's an honest comparison of the four most common, and when to use each.

### Pattern 1: Three-Layer (Primitives → Compositions → Environments)
*Used in this post — recommended for most teams*

```
modules/eks-cluster/    ← single-resource primitive
compositions/aws-stack/ ← wires primitives together
environments/prod/      ← thin config wrapper
```

**Choose when**: You have 5+ environments, multiple teams, or multi-cloud. This gives the best reuse-to-complexity ratio. Changes to a primitive propagate to all environments through compositions.

**Avoid when**: You have only 1–2 environments. The extra indirection adds overhead without benefit.

### Pattern 2: Flat Modules (Single Layer)

```
modules/eks-cluster/
modules/rds-postgres/
modules/vpc-network/
environments/prod/main.tf  ← calls modules directly, no composition layer
```

**Choose when**: Small teams (1–3 engineers), 2–4 environments, simple infrastructure. Easy to understand, fast to iterate on.

**Avoid when**: 10+ environments. You'll end up copy-pasting module calls across environment configs with slight variations.

### Pattern 3: Terragrunt DRY Pattern

```
# terragrunt.hcl per environment — no copy-paste of backend config
terraform {
  source = "git::https://github.com/naik-org/terraform-modules.git//modules/eks-cluster?ref=v1.2.0"
}

inputs = {
  cluster_name = "prod-us-east"
  desired_nodes = 6
}
```

**Choose when**: Large organizations with many environments where even environment-layer boilerplate (backend config, provider versions) becomes repetitive. Terragrunt eliminates all duplication.

**Avoid when**: Your team isn't familiar with Terragrunt. The extra abstraction layer has a learning curve and makes debugging harder.

### Pattern 4: Terraform Registry Modules (Public/Private)

```hcl
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "= 20.14.0"   # Pin exact version

  cluster_name = "prod-cluster"
  # ... inputs
}
```

**Choose when**: Using well-maintained community modules (e.g. `terraform-aws-modules`) for standard resources like VPCs, EKS, RDS. Saves months of work.

**Avoid when**: You need tight control over every resource attribute, or when the community module adds resources you don't want. Forking a public module is worse than writing your own.

### Decision Summary

| Your Situation | Recommended Pattern |
|---|---|
| 1–2 environments, small team | Flat modules |
| 3–10 environments, medium team | Three-layer (primitives → compositions → environments) |
| 10+ environments, large org | Terragrunt DRY pattern |
| Standard AWS/Azure resources | Registry modules + thin wrapper |
| Multi-cloud, complex topology | Three-layer + provider aliasing |

> **My recommendation**: Start with flat modules. Migrate to three-layer when you hit your third environment. Add Terragrunt only when backend config duplication becomes painful — usually around 8–10 environments.

## Key Takeaways

1. **Pin provider versions exactly** — `= 5.55.0`, not `~> 5.0`. Breaking changes in major providers destroy production at 3am.
2. **State is your most critical artifact** — enable versioning, encryption, and access logging on the S3 state bucket.
3. **Use `prevent_destroy` on critical resources** — VPCs, RDS instances, and S3 buckets should never auto-delete.
4. **Modules are contracts** — once a module is consumed by a live environment, treat its input variables like a public API.
5. **Drift is inevitable** — build drift detection into your pipeline, don't rely on memory.
