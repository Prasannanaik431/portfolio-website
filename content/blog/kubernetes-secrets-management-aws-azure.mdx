# Kubernetes Secrets Management at Scale: AWS Secrets Manager + Azure Key Vault + Kyverno

Managing secrets in a multi-tenant Kubernetes environment is one of the hardest Platform Engineering challenges. At Mareana, I designed and implemented a **unified secrets management architecture** spanning AWS and Azure — using native cloud vaults with Kubernetes-native injection, enforced by Kyverno admission policies.

This post covers the full lifecycle: secret creation, automatic rotation, dynamic injection into pods, and policy enforcement to prevent misconfiguration.

---

## The Problem with `kubectl create secret`

The typical workflow teams default to:

```bash
# Anti-pattern: secrets stored in etcd as base64, no rotation
kubectl create secret generic db-credentials \
  --from-literal=DB_PASSWORD=mysupersecretpassword \
  -n production
```

Problems with this approach:
- **Secrets stored in etcd** — exposed if etcd is compromised or backed up to S3
- **No rotation** — credentials age indefinitely
- **No audit trail** — no log of who accessed what
- **Manual lifecycle** — developers must remember to rotate

## Step 1: Install External Secrets Operator

[External Secrets Operator (ESO)](https://external-secrets.io/) syncs secrets from cloud vaults into Kubernetes Secrets:

```bash
# Install ESO via Helm
helm repo add external-secrets https://charts.external-secrets.io

helm upgrade --install external-secrets \
  external-secrets/external-secrets \
  --namespace external-secrets \
  --create-namespace \
  --set installCRDs=true \
  --set webhook.port=9443
```

Verify the operator is running:

```bash
kubectl get pods -n external-secrets

# NAME                                               READY   STATUS
# external-secrets-5d9bdb6c6c-8v2xk                 1/1     Running
# external-secrets-webhook-7f99c6f8b9-ld4pt          1/1     Running
# external-secrets-cert-controller-5f75bc9f4-qwrtz   1/1     Running
```

## Step 2: Configure AWS SecretStore

For AWS, we use IAM Roles for Service Accounts (IRSA) for secure, credential-free authentication:

```yaml
# aws-secretstore.yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-secretstore
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets
```

Create the ExternalSecret that pulls from AWS and creates a K8s Secret:

```yaml
# payment-db-external-secret.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-db-credentials
  namespace: prod-payments
spec:
  refreshInterval: 1h            # Auto-refresh every hour
  secretStoreRef:
    name: aws-secretstore
    kind: ClusterSecretStore
  target:
    name: payment-db-secret      # K8s Secret name created
    creationPolicy: Owner
  data:
    - secretKey: DB_HOST
      remoteRef:
        key: prod/payment-service/database
        property: host
    - secretKey: DB_PASSWORD
      remoteRef:
        key: prod/payment-service/database
        property: password
    - secretKey: DB_USERNAME
      remoteRef:
        key: prod/payment-service/database
        property: username
```

Apply and verify:

```bash
kubectl apply -f payment-db-external-secret.yaml

kubectl get externalsecret payment-db-credentials -n prod-payments
# NAME                     STORE            REFRESH INTERVAL   STATUS   READY
# payment-db-credentials   aws-secretstore  1h                 Valid    True
```

## Step 3: Azure Key Vault SecretStore

For AKS workloads, we use Workload Identity (federated credentials) with Azure Key Vault:

```yaml
# azure-secretstore.yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: azure-keyvault-store
spec:
  provider:
    azurekv:
      vaultUrl: https://mareana-prod.vault.azure.net/
      authType: WorkloadIdentity
      serviceAccountRef:
        name: external-secrets-sa
        namespace: external-secrets
```

ExternalSecret for Azure:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: api-auth-secret
  namespace: prod-api
spec:
  refreshInterval: 30m
  secretStoreRef:
    name: azure-keyvault-store
    kind: ClusterSecretStore
  target:
    name: api-auth-keys
    creationPolicy: Owner
  data:
    - secretKey: JWT_SIGNING_KEY
      remoteRef:
        key: api-jwt-signing-key
    - secretKey: OAUTH_CLIENT_SECRET
      remoteRef:
        key: api-oauth-client-secret
```

## Step 4: Automatic Secret Rotation

AWS Secrets Manager can rotate secrets on a schedule. Configure a Lambda-based rotation:

```bash
# Enable automatic rotation (every 30 days)
aws secretsmanager rotate-secret \
  --secret-id prod/payment-service/database \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123456:function:SecretsManagerRDSPostgreSQLRotationSingleUser \
  --rotation-rules AutomaticallyAfterDays=30
```

With ESO's `refreshInterval: 1h`, the updated secret is automatically synced to the Kubernetes Secret within one hour — **zero manual intervention**.

## Step 5: Enforce via Kyverno Policy

The final piece is preventing teams from bypassing ESO and creating raw `kubectl create secret` objects. We enforce this with Kyverno:

```yaml
# kyverno-require-eso-secrets.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-eso-managed-secrets
  annotations:
    policies.kyverno.io/title: Require ESO-managed Secrets
    policies.kyverno.io/severity: high
    policies.kyverno.io/description: >
      All Kubernetes Secrets must be managed by External Secrets Operator.
      Manually created secrets are not permitted in production namespaces.
spec:
  validationFailureAction: Enforce
  background: false
  rules:
    - name: check-secret-owner
      match:
        any:
          - resources:
              kinds:
                - Secret
              namespaces:
                - prod-*
      exclude:
        any:
          - resources:
              kinds:
                - Secret
              namespaceSelector:
                matchLabels:
                  type: system
      validate:
        message: "Secrets in prod namespaces must be managed by ExternalSecret (owner label required)."
        pattern:
          metadata:
            ownerReferences:
              - apiVersion: external-secrets.io/v1beta1
                kind: ExternalSecret
```

Test the policy:

```bash
# This will be DENIED by Kyverno
kubectl create secret generic test-secret \
  --from-literal=password=abc123 \
  -n prod-payments

# Error from server: admission webhook "validate.kyverno.svc" denied the request:
# Secrets in prod namespaces must be managed by ExternalSecret (owner label required).
```

## Step 6: Monitoring & Alerting

Track ESO sync failures with Prometheus:

```yaml
# prometheus-rule-eso.yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: eso-sync-alerts
spec:
  groups:
    - name: external-secrets
      rules:
        - alert: ExternalSecretSyncFailed
          expr: externalsecrets_sync_calls_total{status="error"} > 0
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "ExternalSecret sync failed"
            description: "Secret {{ $labels.name }} in namespace {{ $labels.namespace }} failed to sync from vault."
```
